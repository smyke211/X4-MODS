# X4Native Versioning

## Version Sources of Truth

| Constant | File | Example | Changes When |
|---|---|---|---|
| `X4NATIVE_API_VERSION` | `sdk/x4native_extension.h` | `1` | ABI-breaking SDK struct change (manual) |
| `X4_GAME_VERSION_LABEL` | `sdk/x4_game_types.h` | `"900-605025"` | New game build (auto-generated by `generate_headers.ps1`) |
| `X4_GAME_TYPES_BUILD` | `sdk/x4_game_types.h` | `900` | New game major version (auto-generated) |
| `X4NATIVE_RELEASE_STR` | CMake define / `x4native_defs.h` fallback | `"r1"` / `"dev"` | Set by CI from tag, defaults to "dev" locally |

## Version Format

```
v9.0.0-605025-r1
 ^^^^^ ^^^^^^  ^^
 │     │       └── release tag (from git tag, set by CI)
 │     └────────── game build number (from X4_GAME_VERSION_LABEL)
 └──────────────── game major.minor.patch (parsed from X4_GAME_TYPES_BUILD "900")
```

## Release Workflow

1. Make changes, commit, push
2. Tag: `git tag v9.0.0-605025-r1 && git push --tags`
3. CI builds, patches `content.xml`, packages, publishes GitHub release

That's it. No source files to edit for releases.

## What CI Does

- Reads `X4_GAME_VERSION_LABEL` and `X4_GAME_TYPES_BUILD` from source
- Extracts release tag from git tag name (`v9.0.0-605025-r2` → `r2`)
- Passes `-DX4NATIVE_RELEASE_STR=r2` to CMake configure
- Patches `content.xml version=` from `X4_GAME_TYPES_BUILD`
- Packages: end-user zip, SDK zip, symbols zip
- Creates GitHub release

## Local Dev Builds

`X4NATIVE_RELEASE_STR` defaults to `"dev"` (no CMake define needed).
Logs show: `X4Native core v900-605025-dev initializing...`
